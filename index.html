Gaps with Existing Persistent Storage Solutions for Peer-Pods

CSI-Wrapper Approach

Wraps existing CSI drivers by intercepting gRPC calls via a proxy. Uses a CRD (PeerpodVolume) to coordinate attachment.

  • High complexity — 3 wrapper binaries per CSI driver
  • Tight coupling — must re-deploy for every CSI driver
  • Custom CRD adds operational overhead & RBAC complexity
  • If the driver changes its NodeID format, the wrapper must be updated
  • Some CSI drivers talk to the K8s API server and cache nodes/PVs — confused by VMs without Node objects
  • Patching existing CSI drivers impacts supportability and maintainability
  • In managed services (e.g. OpenShift), patching CSI drivers is not possible as they are managed entities

Direct Remote FS Mounting

Mount object storage (S3 via Mountpoint, Azure Blob via BlobFuse) directly inside the container using FUSE.

  • Object storage only — not block; no databases or transactional workloads
  • Requires privileged containers (securityContext.privileged: true)
  • Application must manage mounting — not transparent (no standard PVC)
  • Cloud-specific — separate tooling per provider

Gap: No CSI-native, transparent block storage for Peer Pods — that's what our solution fills.

How It Works: Kata Direct Volumes & mountInfo.json

The Existing Kata Protocol

  • Kata Containers defines a "direct volume" protocol — any process can write a mountInfo.json file to a well-known path
  • Path: /run/kata-containers/shared/direct-volumes/<base64(targetPath)>/mountInfo.json
  • Originally built for block device passthrough (Databricks, Ant Group use cases)

What we added to CAA

  • GetCSIVolumesForPod in pkg/util/cloud.go scans this directory
  • Filters by pod UID, reads disk ID & fs type
  • Proxy builds cloud_volumes annotation for PodVM
  • Interceptor handles device detection & mounting inside PodVM

mountInfo.json Contents

// Written by caa-csi-block-driver
{
  "volume-type": "directvol",
  "device": "/subscriptions/.../disks/csi-vol-pvc-abc",
  "fstype": "ext4",
  "metadata": {
    "cloud-provider": "azure",
    "cloud-volume-id": "pvc-abc123",
    "cloud-volume-path": "/subscriptions/.../disks/csi-vol-pvc-abc"
  }
}

Key: The CSI driver writes metadata; the worker node never mounts the volume. Actual mount happens inside the PodVM.

Component 2: CAA Proxy

pkg/adaptor/proxy/service.go PR #3037

  • Scans kataDirectVolumesDir in canonical order for mountInfo.json files
  • Filters volumes by pod UID (io.kubernetes.cri.sandbox-uid) — prevents cross-pod volume leakage
  • Reads disk_id and fs_type from mountInfo.json
  • Assigns deterministic LUN indices matching cloud provider disk attachment order
  • Skipped volumes (no OCI mount) still increment LUN index — ensures consistency
  • Constructs cloud_volumes annotation and forwards to PodVM agent
// cloud_volumes annotation set by the proxy
{
  "vol-0": {
    "mount_point": "/mnt/data",
    "fs_type": "ext4",
    "lun": "0",
    "disk_id": "/subscriptions/.../disks/csi-vol-pvc-abc123"
  }
}

Key design: Canonical directory scan order ensures LUN indices match the order used when attaching disks, preventing mount-to-wrong-disk issues.

Provider-Aware Device Detection

  • Azure — /dev/disk/azure/data/by-lun/N, Hyper-V by-path symlinks, SCSI host 0 filtering
  • AWS — /dev/disk/by-id/nvme-Amazon_Elastic_Block_Store_*
  • Libvirt — /dev/vd[b-z] virtio devices
  • Fallback — SCSI HCTL sysfs LUN matching

Safe formatAndMount

  • Check if already mounted (idempotent)
  • Try mount first — preserves existing data
  • blkid check with retries
  • Only mkfs on genuinely empty disks
  • Never reformats existing filesystems
  • Filesystem type allowlist (ext4, ext3, xfs)

Component 4: Cloud Providers

cloud-providers/azure/provider.go cloud-providers/aws/provider.go PR #3037

Azure — Managed Disks

  • Disks attached during VM creation via getVMParameters
  • LUN assignment matches canonical index from proxy
  • Delete option set to Detach — disk survives VM deletion
  • Device path: /dev/disk/azure/data/by-lun/N
// Azure: attach during VM creation
dataDisks = append(dataDisks, &armcompute.DataDisk{
  Lun: to.Ptr(int32(i)),
  DeleteOption: Detach,
  ManagedDisk: &{ ID: to.Ptr(vol.DiskID) },
})

AWS — EBS Volumes

  • Volumes attached after instance creation via AttachVolume API
  • Parallel attachment with consistent device naming (/dev/xvd[b-z])
  • Multi-disk matching by EBS volume ID, not index
  • Device path: /dev/disk/by-id/nvme-Amazon_Elastic_Block_Store_*
// AWS: attach after instance launch
for i, vol := range csiVolumes {
  ec2Client.AttachVolume(&ec2.AttachVolumeInput{
    Device: ebsDeviceName(i),
    VolumeId: vol.DiskID,
    InstanceId: instanceID,
  })
}

Persistence model: Cloud disks outlive the PodVM on both providers. When a new pod claims the same PVC, a new VM is created and the same disk is re-attached — data survives across pod restarts.

End-to-End Call Flow

Volume Provisioning

1PVC created with caa-block-azure / caa-block-aws StorageClass
2External provisioner calls CSI driver CreateVolume
3CSI driver provisions Azure Managed Disk / AWS EBS volume
4PV created and bound to PVC

Node Publish

5Pod scheduled → kubelet calls NodePublishVolume
6CSI driver writes mountInfo.json (no mount on host)

CAA & PodVM

7CAA proxy reads mountInfo.json, builds cloud_volumes
8Cloud provider attaches disk to PodVM (Azure: during creation, AWS: after launch)
9PodVM boots → annotation forwarded to interceptor
10Interceptor detects device by LUN via sysfs/by-path
11Format (if empty) → mount → bind-mount into container

Result

12Container sees /mnt/data — persistent block volume

Security & Safety

Pod-Level Volume Isolation

Volumes filtered by io.kubernetes.cri.sandbox-uid — a PodVM can never access another pod's managed disk.

Path Traversal Prevention

Volume names sanitized with filepath.Base() — names like ../../etc are rejected before mount point construction.

Filesystem Type Allowlist

Only ext4, ext3, xfs allowed. Prevents arbitrary command execution via mkfs.<attacker-input>.

Idempotent Mounting

Checks mountinfo.Mounted() before format/mount. Prevents double-mount errors and accidental reformatting.

Safe Format Logic

Mount-first approach: tries mounting before formatting. Existing data is never overwritten. Only truly empty disks get mkfs.

Integer Bounds Checking

Explicit bounds checks before int to int32 narrowing conversions. Prevents overflow in LUN indexing (CodeQL validated).
